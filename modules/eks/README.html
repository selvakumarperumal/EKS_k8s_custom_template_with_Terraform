resource "aws_kms_key" "eks" {
  description             = "KMS key for EKS cluster ${var.cluster_name} encryption"
  deletion_window_in_days = 7    # 7-day grace period before permanent deletion
  enable_key_rotation     = true # Auto-rotate key material annually

  tags = merge(
    var.tags,
    {
      Name = "${var.cluster_name}-eks-kms"
    }
  )
}
📥 Inputs Used
var.cluster_name="eks-prod-us-east-1"
var.tags={ Environment = "production" }
What This Creates in AWS
🔑 KMS Key
→ encrypts →
K8s Secrets in etcd
deletion_window = 7 days
key_rotation = annual
symmetric AES-256-GCM
☸️ Kubernetes Purpose

Without KMS, Kubernetes secrets (DB passwords, API keys) are stored base64-encoded in etcd — NOT encrypted. This key enables envelope encryption: EKS encrypts each secret with a unique data key, then encrypts that data key with this KMS key. Even with etcd access, secrets remain unreadable.

🚀 What Happens When You Apply
PLAN
Terraform shows + aws_kms_key.eks will be created
APPLY
AWS creates a 256-bit symmetric encryption key in KMS. Takes ~2 seconds. Key ID generated automatically.
RESULT
Key ARN like arn:aws:kms:us-east-1:123456:key/abc-123 — used by the EKS cluster resource later.
COST
$1/month per KMS key + $0.03 per 10,000 API calls. Very cheap.
📖 7-Day Deletion Grace Period

When you terraform destroy, AWS doesn't delete the key immediately. It schedules deletion for 7 days later. During this time:

  • Key is disabled — can't encrypt/decrypt anything
  • Any service still using it gets KMSKeyDisabledException
  • You can cancel deletion and restore the key if needed
  • After 7 days → permanently destroyed, all data encrypted with it becomes unrecoverable

resource "aws_security_group" "cluster" {
  name_prefix = "${var.cluster_name}-cluster-sg-" # Unique name with random suffix
  description = "Security group for EKS cluster control plane"
  vpc_id      = var.vpc_id # Place in the same VPC as the cluster

  # EGRESS: Allow ALL outbound traffic from the cluster
  egress {
    from_port   = 0             # All ports
    to_port     = 0             # (0,0 with protocol -1 = all traffic)
    protocol    = "-1"          # All protocols (-1 = any)
    cidr_blocks = ["0.0.0.0/0"] # To any destination
    description = "Allow all outbound traffic"
  }

  tags = merge(
    var.tags,
    {
      Name = "${var.cluster_name}-cluster-sg"
    }
  )

  lifecycle {
    create_before_destroy = true # Avoid downtime during SG replacement
  }
}

Attached to the ENIs that EKS creates in your VPC. These ENIs are how the control plane (API server) communicates with your worker nodes. Without proper SG rules, nodes can't register with the API server and pods can't be scheduled.

# RULE 1: Nodes → Cluster API (HTTPS/443)
resource "aws_security_group_rule" "node_to_cluster" {
  type                     = "ingress"                     # Inbound rule on the cluster SG
  from_port                = 443                           # HTTPS port
  to_port                  = 443                           # Single port
  protocol                 = "tcp"                         # TCP only
  security_group_id        = aws_security_group.cluster.id # Target: cluster SG
  source_security_group_id = aws_security_group.node.id    # Source: node SG
  description              = "Allow nodes to communicate with cluster API"
}

# RULE 2: Cluster API → Nodes (1025-65535)
resource "aws_security_group_rule" "cluster_to_node" {
  type                     = "ingress"
  from_port                = 1025
  to_port                  = 65535
  protocol                 = "tcp"
  security_group_id        = aws_security_group.node.id    # Target: node SG
  source_security_group_id = aws_security_group.cluster.id # Source: cluster SG
  description              = "Allow cluster control plane to communicate with nodes"
}

# RULE 3: Node → Node (All ports/protocols)
resource "aws_security_group_rule" "node_to_node" {
  type              = "ingress"
  from_port         = 0     # All ports
  to_port           = 65535 # All ports
  protocol          = "-1"  # All protocols (TCP, UDP, ICMP)
  security_group_id = aws_security_group.node.id
  self              = true  # Traffic from same SG is allowed
  description       = "Allow nodes to communicate with each other"
}
🚀 What Happens When You Apply
APPLY
Creates 3 security group rules. All are inbound (ingress) rules on their target SGs.
RESULT
Nodes can talk to API server (443) ✓ API server can reach nodes (1025–65535) ✓ Nodes can talk to each other ✓
IF MISSING
Without Rule ①: nodes can't register → kubectl get nodes returns empty. Without Rule ②: kubectl exec and kubectl logs fail.
⚠️ Why separate resources? Defining SG rules as separate aws_security_group_rule resources (instead of inline) prevents circular dependency between the cluster and node SGs. Terraform can create both SGs first, then wire the rules.

Concept
"Security Group vs NACL"
Understanding Stateful vs Stateless network boundaries
Example Concept 
# ┌─────────────────────────────────────────────────────────┐
# │        NACL vs Security Group — Side by Side           │
# └─────────────────────────────────────────────────────────┘

# ── Security Group (Stateful) ──────────────────────────────
# Only 1 rule needed — return traffic is AUTOMATIC
resource "aws_security_group_rule" "allow_http" {
  type              = "ingress"
  from_port         = 80
  to_port           = 80
  protocol          = "tcp"
  cidr_blocks       = ["0.0.0.0/0"]
  security_group_id = aws_security_group.web.id
}
# ✅ Response on port 52341 → auto-allowed (stateful)

# ── NACL (Stateless) ──────────────────────────────────────
# BOTH directions must be explicitly defined!
resource "aws_network_acl_rule" "inbound_http" {
  network_acl_id = aws_network_acl.web.id
  rule_number    = 100
  protocol       = "tcp"
  rule_action    = "allow"
  egress         = false         # inbound
  cidr_block     = "0.0.0.0/0"
  from_port      = 80
  to_port        = 80
}

resource "aws_network_acl_rule" "outbound_ephemeral" {
  network_acl_id = aws_network_acl.web.id
  rule_number    = 100
  protocol       = "tcp"
  rule_action    = "allow"
  egress         = true          # outbound
  cidr_block     = "0.0.0.0/0"
  from_port      = 1024          # ephemeral ports
  to_port        = 65535         # client's random return port
}
# ⚠️ Without the outbound rule, response is SILENTLY DROPPED

# ── Blocking a Specific IP (only NACL can do this) ────────
resource "aws_network_acl_rule" "block_attacker" {
  network_acl_id = aws_network_acl.web.id
  rule_number    = 50            # evaluated BEFORE rule 100
  protocol       = "-1"
  rule_action    = "deny"        # ← SGs cannot do this!
  egress         = false
  cidr_block     = "192.168.1.50/32"  # bad actor
}

💡 Note: Security Groups are deny by default — no matching allow rule = traffic dropped. You can only ADD allow rules, never deny rules. That's why NACLs exist: to block specific IPs.
⚠️ Warning: With NACLs, ALWAYS open ephemeral ports (1024-65535) on the outbound side, otherwise responses never reach the client even though inbound rules look correct.
📖 NACL vs 🛡️ Security Group — The Complete Guide
Feature NACL Security Group
Operates at Subnet level Instance level
State Stateless Stateful
Rules Allow + Deny Allow only
Rule evaluation In order (lowest # first) All rules evaluated together
Default Allow all Deny all inbound

Stateful vs Stateless — The Key Concept:

Security Group (Stateful): If you allow inbound traffic on port 80, the return traffic on port 52341 is automatically allowed. The SG remembers the connection.
NACL (Stateless): You must explicitly allow BOTH directions. The NACL has no memory of connections — every packet is evaluated independently.

Return Traffic Example:
Browser (203.0.113.10:52341) → EC2 (54.23.45.67:80) = request
EC2 (54.23.45.67:80) → Browser (203.0.113.10:52341) = return traffic

The browser picked a random ephemeral port (52341). The response goes back to that port. With SG, this is auto-allowed. With NACL, you need an outbound rule for ports 1024-65535 or the response is silently dropped — the browser just spins forever.

When to use which:
• NACL: When you need to block a specific IP/range (e.g., banning a malicious IP)
• Security Group: For everything else — controlling ports/protocols. Simpler because you don't think about return traffic
• Both must allow traffic for it to reach your instance. Think of NACL as building security and Security Group as the door lock on your apartment.

📖 access_config — Modern Authentication

authentication_mode = "API_AND_CONFIG_MAP" supports two methods:

  • EKS Access Entries (new): Managed via Terraform/CLI. More auditable, no ConfigMap drift.
  • aws-auth ConfigMap (legacy): Manual K8s ConfigMap mapping IAM → RBAC. Still widely used.

☸️ Why IRSA Matters

Without IRSA: All pods share the node role → a compromised pod can access ECR, modify ENIs, call any API. With IRSA: Each pod gets only the permissions it needs. An S3 uploader can write to S3 but can't touch DynamoDB.

Concept
"Launch Template vs Node Group"
The Recipe vs The Bakery
The Division of Labor 
# ┌─────────────────────────────────────────────────────────┐
# │             Launch Template vs Node Group               │
# └─────────────────────────────────────────────────────────┘

# ── 1. The Launch Template (The "Recipe") ──────────────────
# Defines WHAT a single node looks like when it boots up.
resource "aws_launch_template" "eks_nodes" {
  name = "node-template"
  
  # The hardware & OS
  image_id      = "ami-12345ABCD"      # Which OS to run
  instance_type = "t3.medium"          # How much CPU/RAM
  
  # The Disks
  block_device_mappings {
    ebs {
      volume_size = 50                 # 50GB Hard Drive
    }
  }
  
  # The Network Security
  vpc_security_group_ids = [aws_security_group.node.id]
}


# ── 2. The Node Group (The "Manager / Bakery") ─────────────
# Defines HOW MANY nodes to run and WHERE to put them.
# It uses the Launch Template to actually create them.
resource "aws_eks_node_group" "main" {
  cluster_name    = aws_eks_cluster.main.name
  node_group_name = "standard-workers"
  
  # Where to put the nodes
  subnet_ids = ["subnet-A", "subnet-B", "subnet-C"]
  
  # How many nodes to bake
  scaling_config {
    desired_size = 3    # Start with 3
    min_size     = 3    # Never go below 3
    max_size     = 7    # Never go above 7
  }

  # Connects the manager to the recipe!
  launch_template {
    id      = aws_launch_template.eks_nodes.id
    version = aws_launch_template.eks_nodes.latest_version
  }
}

Why are they separate?

In AWS, a Node Group is technically a wrapper around an underlying Auto Scaling Group (ASG). By design, ASGs don't know anything about disks or operating systems. They only know how to scale up and scale down based on a metric (like CPU usage).

To do this, the ASG needs a Launch Template to hand to the AWS EC2 service saying, "Hey, my scale went from 3 to 4. Please spin up an EC2 instance that looks exactly like this template."

1. The Launch Template is completely standalone. It just sits in AWS as a saved text file describing a server.
2. The Node Group tells EKS to create an ASG, points that ASG to your Launch Template, and links the instances securely so they register as Kubernetes worker nodes using the aws_iam_role.

⚠️ Why are instance_types and capacity_type in the Node Group?
While technically you can set an instance type in a Launch Template, AWS and EKS explicitly forbid you from setting it in both places. It is best practice to keep it in the Node Group for EKS because:
  • Cluster Autoscaler: Needs to know the instance type to calculate available CPU/memory for scaling decisions.
  • SPOT Interruption Handling: EKS actively manages spot interruption notices to gracefully drain nodes, requiring it to explicitly know which node groups are using SPOT capacity.
  • Instance Diversity: Node groups support passing a list (e.g., ["t3.large", "t3a.large"]) for SPOT capacity diversity to avoid stockouts. Launch templates only accept one single type.
Simple rule: If EKS actively manages it or makes decisions based on it (scaling, capacity type), it goes in the Node Group. If EKS doesn't care (disks, OS, IMDSv2 security), it goes in the Launch Template.
Real World Scenario: Changing Disk Size
If you change volume_size = 50 to 100:
1. Terraform creates a new version of the Launch Template (e.g., v2).
2. Terraform updates the Node Group to point to v2 instead of v1.
3. The Node Group gracefully spins up new 100GB nodes and cordons/drains the old 50GB nodes one by one (Rolling Update) with zero downtime.

The Solution: The for_each Loop 
# ┌─────────────────────────────────────────────────────────┐
# │             The Dynamic for_each Approach               │
# └─────────────────────────────────────────────────────────┘

# 1. You pass in a map (dictionary) of variables from outside
# variable "node_groups" = {
#   general = { instance_types = ["t3.medium"], capacity = "ON_DEMAND" }
#   batch   = { instance_types = ["t3a.medium"], capacity = "SPOT" }
# }

# 2. You write ONE block of Terraform code
resource "aws_eks_node_group" "main" {
  
  # 3. Terraform loops over the map keys ("general" and "batch")
  for_each = var.node_groups

  # 4. each.key is the name (e.g., "general")
  node_group_name = each.key
  
  # 5. each.value accesses the nested data (e.g., ["t3.medium"])
  instance_types = each.value.instance_types
  capacity_type  = each.value.capacity
}
✨ How it works dynamically
When Terraform sees for_each = var.node_groups, it looks at the map you provided. If the map has 2 items ("general" and "spot"), Terraform dynamically creates 2 distinct resources in AWS.

In your Terraform State (and when you run commands), they are referenced using array bracket notation with the key name:
aws_eks_node_group.main["general"]
aws_eks_node_group.main["spot"]

The Magic: You never have to touch this main.tf file again to add a new Node Group. You just add a new 3-line entry to your terraform.tfvars variables file, and Terraform automatically spins up the new group using this exact blueprint!

📖 Why IMDSv2 Matters (Capital One Breach)

In 2019, a Capital One breach exploited IMDSv1 through SSRF. The attacker made the server request http://169.254.169.254/latest/meta-data/iam/ and stole IAM credentials. IMDSv2 prevents this by requiring a PUT request for a token first — SSRF attacks typically only support GET.

http_tokens = "required" = IMDSv1 is completely blocked.

resource "aws_eks_node_group" "main" {
  for_each = var.node_groups # One node group per entry

  cluster_name    = aws_eks_cluster.main.name # Which cluster to join
  node_group_name = each.key                  # "general" or "spot"
  node_role_arn   = var.node_role_arn         # IAM role for nodes
  subnet_ids      = var.subnet_ids            # Private subnets
  version         = var.kubernetes_version    # K8s version for nodes

  # ── Scaling Configuration ──
  scaling_config {
    desired_size = each.value.desired_size # Starting count
    max_size     = each.value.max_size     # Autoscaler ceiling
    min_size     = each.value.min_size     # Autoscaler floor
  }

  instance_types = each.value.instance_types # e.g., ["t3.medium"]
  capacity_type  = lookup(each.value, "capacity_type", "ON_DEMAND")

  # Kubernetes labels applied to all nodes in this group
  labels = lookup(each.value, "labels", {})

  # Dynamic taint blocks — generates 0 or more taint configurations
  dynamic "taint" {
    for_each = coalesce(lookup(each.value, "taints", null), [])
    content {
      key    = taint.value.key    # e.g., "spot"
      value  = taint.value.value  # e.g., "true"
      effect = taint.value.effect # e.g., "NO_SCHEDULE"
    }
  }

  # Use our custom launch template
  launch_template {
    id      = aws_launch_template.node[each.key].id             # Template ID
    version = aws_launch_template.node[each.key].latest_version # Always latest
  }

  # Merge common tags with node-group-specific tags
  tags = merge(
    var.tags,
    lookup(each.value, "tags", {})
  )

  # Needs VPC CNI and kube-proxy ready before nodes join
  depends_on = [
    aws_eks_addon.vpc_cni,
    aws_eks_addon.kube_proxy
  ]

  # Ignore desired_size — Cluster Autoscaler manages this at runtime
  lifecycle {
    ignore_changes = [scaling_config[0].desired_size]
  }
}
🧮 How Many Nodes Will Be Created?

The for_each = var.node_groups creates one node group per map entry. With the typical config:

Node Group: "general" (ON_DEMAND)
desired_size = 2 ← starts with 2 nodes
min_size = 2 ← never goes below 2
max_size = 4 ← autoscaler ceiling
instance = t3.medium (2 vCPU, 4 GiB RAM)
Node Group: "spot" (SPOT)
desired_size = 1 ← starts with 1 node
min_size = 1 ← never goes below 1
max_size = 3 ← autoscaler ceiling
instance = t3.medium, t3a.medium
Initial: 2 + 1 = 3 nodes  |  Min: 2 + 1 = 3  |  Max: 4 + 3 = 7 nodes
At first terraform apply, exactly 3 EC2 instances are launched.
💡 What happens if I pass multiple instance types?

Passing multiple instance types (e.g., instance_types = ["t3.large", "t3a.large", "m5.large"]) behaves very differently depending on your capacity type.

SPOT: Diversification (Recommended)
AWS picks from this list based on exactly what is cheapest and most available at that second. If t3.large has no capacity in your AZ, it seamlessly tries t3a.large, then m5.large.

The Golden Rule: Provide 4-6 similar sized types (same vCPU/RAM). More instance types = less chance of interruptions or failing to scale!
ON_DEMAND: First Available
ON_DEMAND is practically always available, so AWS simply goes in order and picks the first one on your list.

The Golden Rule: You usually only need to pass 1 (maybe 2) instance types for ON_DEMAND. Passing a huge list doesn't provide the same diversification benefits as SPOT.
⚠️ Will my node run ALL of these types at once?
No! At the end of the day, one instance type is launched per node. The list just tells AWS: "Here are your options, pick the best one available right now."
⚙️ How Node Creation Actually Works (Under the Hood)

When Terraform creates an aws_eks_node_group, here's the exact sequence:

①
Terraform → EKS API
Terraform calls CreateNodegroup with your scaling config, instance types, and launch template.
②
EKS → Auto Scaling Group
EKS creates an AWS Auto Scaling Group (ASG) behind the scenes. You don't see this in your Terraform — EKS manages it internally. The ASG uses your launch template.
③
ASG → EC2 Instances
ASG launches desired_size EC2 instances (e.g., 2 for "general"). Instances are spread across your subnets for multi-AZ availability.
④
EC2 → Bootstrap Script
The EKS-optimized AMI runs a bootstrap script (/etc/eks/bootstrap.sh) that:
  • Configures kubelet with the cluster API endpoint and CA certificate
  • Sets the --node-labels from your labels config
  • Applies --register-with-taints from your taints config
  • Starts kubelet as a systemd service
⑤
kubelet → API Server
kubelet connects to the API server (via the cluster SG rule #6①) and creates a Node object. The node status transitions: NotReady → Ready.
⑥
VPC CNI → ENIs
The VPC CNI DaemonSet allocates secondary IP addresses on the node's network interface. These IPs become available for pods. A t3.medium supports ~17 pod IPs per ENI.
⑦
Scheduler → Pods
The scheduler notices the new Ready node and starts placing pending pods on it. kube-proxy DaemonSet and any DaemonSets automatically deploy a pod to the new node.
Node Bootstrap Timeline
t=0s : ASG launches EC2 instance
t=30s: Instance booting, bootstrap.sh starts
t=45s: kubelet starts, connects to API server
t=60s: Node registers, status = NotReady
t=90s: VPC CNI allocates IPs, kube-proxy starts
t=120s: Node status = Ready ✅ — pods can be scheduled
💰 Spot vs On-Demand — Why Two Node Groups?
ON_DEMAND ("general")
  • Guaranteed availability — never interrupted
  • Full price: ~$0.0416/hr for t3.medium
  • For: databases, API servers, critical workloads
  • No taints — any pod can schedule here
SPOT ("spot")
  • Can be interrupted with 2-min warning
  • 60-90% cheaper: ~$0.0125/hr for t3.medium
  • For: batch jobs, CI/CD, stateless workers
  • Tainted with spot=true:NoSchedule
🚧 Taints & Tolerations — How Spot Nodes Repel Normal Pods

The dynamic "taint" block in the HCL applies Kubernetes taints to spot nodes:

# Example taint on the "spot" node group: taints = [{ key = "spot" value = "true" effect = "NO_SCHEDULE" # Pods without toleration are rejected }]
📦 Normal Pod (no toleration)
→ tries to schedule on spot node →
❌ REJECTED
📦 Batch Pod (with toleration)
→ tries to schedule on spot node →
✅ ACCEPTED
# Deployment that can run on spot nodes: tolerations: - key: "spot" operator: "Equal" value: "true" effect: "NoSchedule"
📏 Scaling — Why lifecycle { ignore_changes = [desired_size] }?
Cluster Autoscaler
→ "need more pods, scaling to 4" →
desired_size = 4
terraform apply
→ "desired_size should be 2" →
❌ Scales BACK to 2!

Without ignore_changes, every terraform apply fights the autoscaler, resetting node count to the original value. The lifecycle block tells Terraform: "Don't touch desired_size — the autoscaler manages that at runtime."
